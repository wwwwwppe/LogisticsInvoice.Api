# 面试讲法

## 3 分钟项目介绍

我在做一个 `LogisticsInvoice.Api` 学习项目，定位是物流发票与报表管理后端。这个选题不是凭空想的，因为我早期参与过 .NET Framework、ASP.NET、Entity Framework 和 Oracle 的 ERP 系统开发维护，后来也维护过 Java 企业业务系统，接触过对账、结算、发票、审批流和外部接口。

我做这个项目的目的，是把过去偏维护型、传统框架下的业务经验，用现代 ASP.NET Core Web API 的方式重新实现。项目当前采用 .NET 6、ASP.NET Core Web API、EF Core、Oracle Database XE 21c 和 Swagger，Oracle 通过 Docker 提供一致的本地环境。项目先做结构清楚的单体应用，没有一开始引入微服务、CQRS 或消息队列。

第一阶段完成的是基础骨架和客户/供应商主数据模块。我把客户和供应商抽象成业务往来单位 `BusinessParty`，用类型区分客户、供应商或二者兼有。这是因为它们当前字段相似，而且现实中同一公司可能同时扮演两种角色。接口支持新增、修改、条件分页查询和启用停用。

代码按 Controller、Service、Repository 分工：Controller 处理 HTTP 和状态码，Service 放重复编码校验、分页限制、状态修改等业务规则，Repository 用 EF Core 访问数据库。接口层使用 DTO，避免直接暴露数据库实体。另外做了统一响应结构、参数校验和全局异常处理中间件。业务异常会给调用方明确提示，未知异常只返回通用错误和 traceId，详细堆栈留在服务端日志。

下一阶段我会把自动建库切换成 Migration，然后实现发票主表、明细和状态流转，再加入 JWT 角色权限、操作日志、统计报表和 Excel。我的重点不是堆技术名词，而是把一个企业业务系统从数据建模、接口、异常、权限到可追踪性完整走一遍，并且能解释每个选择。

## 技术亮点

### 1. 保持简单但职责清晰

- 单体 Web API，适合一个月学习周期。
- Controller、Service、Repository 各自职责明确。
- 没有为了面试强行引入微服务、DDD、CQRS、Kafka、Redis 或 Kubernetes。

### 2. 接口契约稳定

- 使用输入 DTO 和输出 DTO，不直接暴露 EF Core Entity。
- Data Annotation 做基础参数校验。
- `ApiResponse<T>` 统一 `success`、`message`、`data`、`errors`、`traceId`。
- HTTP 状态码仍表达请求结果，不是所有错误都返回 200。

### 3. 异常集中治理

- 业务错误抛出 `BusinessException`，由中间件转为统一响应。
- 未知异常记录日志，不向客户端泄露堆栈。
- `traceId` 用于关联客户端错误和服务端日志。

### 4. EF Core 查询细节

- 列表只读查询使用 `AsNoTracking`，减少不必要的变更跟踪。
- 分页前先按条件统计总数，再 `Skip/Take` 获取当前页。
- 对编码建立唯一索引，并在 Service 先给出友好的重复提示。
- 数据访问使用异步 API，避免 Web 请求线程被同步数据库调用长期占用。

### 5. Oracle 开发环境可复现

- 使用 Oracle 官方 `Oracle.EntityFrameworkCore 7.21.13` Provider 和纯托管 ODP.NET 驱动。
- Docker Compose 固定 Oracle Database XE 21c 官方镜像、端口、幂等启动脚本和数据卷。
- 应用使用独立 `logistics_invoice` Schema，不使用 `SYS` 或 `SYSTEM` 承载业务表。
- 本地开发配置可以开箱运行，真实环境通过环境变量覆盖连接字符串。

## 业务亮点

- 客户和供应商统一建模为业务往来单位，支持同一主体兼具两种角色。
- 编码作为稳定业务标识，创建后暂不允许通过修改接口改变。
- 停用代替物理删除，便于后续保留发票历史关系和审计链路。
- 分页支持名称/编码关键词、单位类型和启用状态组合筛选。
- 为后续发票归属、结算对象、报表维度预留统一主数据基础。

## 常见追问与回答

### 为什么用 .NET 6，而不是最新版本？

这个项目的学习目标明确指定 .NET 6，也贴近一些企业存量系统的技术环境。我知道 .NET 6 已结束官方支持；如果用于新生产项目，我会优先选择仍在支持期的 LTS 版本。这里先完成既定学习目标，同时把升级作为后续可演示项。

### 为什么使用 Oracle Entity Framework Core 7.21.13？

项目目标框架仍是 `net6.0`。Oracle 官方文档说明 EF Core 7 Provider 支持 .NET 6 和 7，并支持连接 Oracle Database 12.2 以上版本。7.21.13 属于 EF Core 7 兼容线，适合当前技术约束；这不代表应用目标框架变成了 .NET 7。

### 为什么从 SQL Server LocalDB 改为 Oracle？

SQL Server LocalDB 的本地启动成本更低，但 Oracle 与我过去维护 ERP、发票、报表和企业接口的真实经历更一致。这个选择能让我继续练习 Oracle Schema、数据类型、SQL 和后续 PL/SQL 场景。它不是抽象意义上的“Oracle 一定更好”，而是对当前求职叙事和学习目标更合适。

### 为什么 Oracle 放在 Docker，而不是直接安装？

开发机没有可用的 Oracle 实例。Docker Compose 可以固定数据库版本、端口、应用用户、健康检查和数据卷，其他人克隆仓库后也能复现。代价是镜像较大、首次启动较慢，所以 README 明确要求先等数据库健康再启动 API。

### 应用为什么不用 `SYS` 或 `SYSTEM` 账号？

管理员账号权限过大，不应该承载应用业务表。容器初始化时创建独立的 `logistics_invoice` 用户，它同时拥有自己的 Schema，应用连接只使用这个账号。后续还可以继续收紧权限。

### 切换 Provider 只改 `UseSqlServer` 为 `UseOracle` 就够了吗？

当前简单模型的 LINQ 和字段类型都能由 Provider 转换，所以代码变化不大；但不能推导出所有数据库都能无成本切换。真实项目还必须验证类型映射、分页、索引、大小写、迁移脚本、原生 SQL、事务隔离和执行计划。本项目已经用真实 Oracle 容器跑接口，而不是只证明能编译。

### 为什么不拆成客户表和供应商表？

当前字段高度重合，同一主体也可能既是客户又是供应商。统一表减少重复，并让后续发票和结算引用同一主数据。如果后期专属字段明显增加，我会增加类型扩展表，而不是继续把大量可空字段塞进主表。

### 为什么还要 Repository？EF Core 的 DbContext 本身就像 Repository。

这是合理追问。当前项目保留一层很薄的 Repository，是为了让数据查询和业务规则位置更容易学习和展示。它没有再抽象通用仓储，也没有隐藏 `IQueryable` 做复杂框架。如果模块很简单或团队直接使用 DbContext，我也接受省略 Repository，关键是不要机械套层。

### 为什么不用 AutoMapper？

当前 DTO 数量少，手工映射更直接，也能清楚看到哪些字段对外暴露。映射大量重复后再评估 AutoMapper，避免为了少量代码引入额外约定和调试成本。

### 统一响应和 HTTP 状态码会不会重复？

二者解决不同问题。HTTP 状态码让网关、监控和通用客户端判断结果；统一响应给业务调用方稳定的消息、数据、错误详情和 traceId。比如未找到返回 HTTP 404，同时响应体仍符合 `ApiResponse<T>`。

### 为什么不用全局 `try/catch` 写在 Controller？

每个 Controller 重复捕获会产生模板代码，也容易漏掉。Middleware 位于请求管道中，能统一捕获后续组件异常，集中决定状态码、日志和响应格式。

### `EnsureCreated` 能用于生产吗？

不建议。它适合第一阶段从零创建开发数据库，但不管理持续演进的数据库版本。下一步会使用 EF Core Migration 生成可审查的变更，并讨论生产发布、备份和回滚。

### 如何避免两个请求同时创建相同编码？

Service 的存在性检查能提供友好提示，但不能独自解决并发竞争，所以数据库还建立了唯一索引。完整实现还应捕获唯一键冲突并转换为业务错误。这个并发边界可以作为后续增强项，不能声称当前已经完全处理。

### 为什么停用而不是删除？

客户或供应商一旦被发票引用，物理删除会破坏历史关系。停用可以阻止新业务继续选择，同时保留历史数据。后续会在发票创建时校验往来单位必须启用。

## 如何说明“这是学习项目，但和真实经历有关”

推荐说法：

> 这是我为了补强现代 .NET 后端能力主动做的学习项目，不是原公司的生产系统。我没有复制原公司的代码和数据，但选题、字段取舍和流程问题来自我过去维护 ERP、发票、对账、结算和报表系统时积累的业务理解。我用 ASP.NET Core 和 EF Core 重新实现，是为了证明我能把既有企业业务经验迁移到现代 .NET 技术栈。

不要说：

- “这是我给某大型客户上线的物流发票平台。”
- “系统支撑了多少万用户或多少亿金额。”当前没有证据。
- “采用企业级高并发架构。”当前没有压测和生产运行。
- “完整实现了权限、审计和报表。”这些仍在后续计划中。

## 第一阶段完成后，面试者应该能讲清楚

1. 一个请求从 Controller 到 Service、Repository、DbContext 的调用路径。
2. Entity 和 DTO 为什么分离。
3. EF Core 跟踪查询与 `AsNoTracking` 的差别。
4. 分页查询的 `Count + Skip + Take`。
5. 业务异常与未知异常为什么分开处理。
6. HTTP 状态码与统一返回结构如何配合。
7. 当前方案哪些只是开发便利，哪些可以延续到生产设计。
